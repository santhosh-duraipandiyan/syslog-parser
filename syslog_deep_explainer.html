The simplest explanation
Imagine every program on your computer keeps a diary. Every time something important happens — a user logs in, a file is created, a network connection is made — the program writes one line in its diary. Syslog is the standardised format for writing those diary entries, and the system for collecting all those diaries in one place.
Why it matters in blue team work
Detection
Every brute force attack, every lateral movement, every privilege escalation — they all leave syslog entries. Your job is to find the suspicious ones before the attacker achieves their goal.
Forensics
After an incident, syslog is how you reconstruct "what happened and when". It's your timeline. Without it you're blind to what the attacker did inside your network.
Correlation
A failed SSH login on one machine means little. The same IP failing 200 times across 30 machines in 60 seconds means something very different. Correlation requires parsing.
SIEM feeding
Tools like Splunk, Microsoft Sentinel, and Elastic SIEM all consume syslog. The better your parser, the richer the data your SIEM gets to work with.
Where syslogs come from
Every device that speaks TCP/IP can send syslogs. They all look slightly different, which is exactly why a parser is needed.
Linux servers — /var/log/auth.log, syslog
Windows — Event Log forwarded via NXLog
Cisco routers & switches — IOS mnemonics
Cisco firewalls — ASA connection logs
Palo Alto firewalls — CSV traffic/threat logs
AWS — VPC Flow Logs via CloudWatch
Web servers — Apache/Nginx access logs
Applications — any custom syslog sender
The anatomy of every syslog line
PRI timestamp hostname app / process PID message 
<34>Oct 11 22:14:15 webserver01 sshd[29785]: Failed password for alice from 192.168.1.42 port 22 ssh2
Every syslog line — no matter which platform it comes from — tries to tell you the same five things: when it happened (timestamp), where it came from (hostname), what generated it (app/process), how bad it is (severity inside PRI), and what happened (message).
What the PRI field is
The number inside <angle brackets> at the start of every syslog message is called the Priority (PRI). It looks like just one number, but it secretly contains two pieces of information squeezed together using simple maths.
The formula
Facility = what part of the system generated the log.

Examples: 0=kernel, 4=auth, 10=authpriv, 16-23=local0-7

Severity = how bad the event is, on a scale of 0–7.

0=emergency, 1=alert, 2=critical, 3=error, 4=warning, 5=notice, 6=info, 7=debug
PRI = (facility × 8) + severity To decode PRI 34: facility = 34 ÷ 8 = 4 → "auth" severity = 34 mod 8 = 2 → "critical" In Python (from the code): facility_num = pri_val >> 3 # same as ÷ 8 severity_num = pri_val & 0x07 # same as mod 8

What the alert engine does
After every log line is parsed into a structured object, the alert engine checks it against a list of rules. Each rule is just a Python function that returns True or False. If it returns True, the alert fires.
How a rule works — step by step
rule = {
    "id":   "AUTH001",
    "name": "SSH brute force — failed password",

    # This is a lambda — a tiny one-line function.
    # It receives the parsed log object (l) and checks
    # two things at once using 'and':
    "check": lambda l: (
        l.action == "auth_failed"       # did parse_ssh_message() set this?
        and
        l.app and "sshd" in l.app.lower()  # is the app actually SSH?
    ),
    "severity": "high",
}
A lambda is just a function written in one line. lambda l: l.action == "auth_failed" means "given a log object called l, return True if its action field equals auth_failed". Same as writing a def function but shorter.
The rules in the code and what they catch
AUTH001 — SSH brute force
Fires when action == "auth_failed" AND app contains "sshd". The parse_ssh_message() function sets the action field by looking for "Failed" in the message text.
AUTH002 — Root login attempt
Fires when username == "root" AND app is sshd. Note that this fires even on successful root logins — any SSH connection targeting root is suspicious in most environments.
PRIV001 — Sudo escalation to root
Fires when app contains "sudo" AND extra["sudo_target"] == "root". The parse_sudo_message() function extracts the target user from the TTY/PWD/USER/COMMAND line format.
WIN001 / WIN002 — Windows EventIDs
Checks event_id == "4625" (failed logon) or "4720" (account created). These are extracted from the Windows syslog bridge format message body using parse_windows_message().
In a real SIEM, rules are loaded from YAML or JSON config files — not hardcoded. You'd also add a counter (how many AUTH001s from the same IP in 60 seconds?) and a threshold (fire a higher-severity alert at 5+). That's the foundation of SIEM correlation.

How a SOC analyst uses a syslog parser
A parser on its own doesn't protect anything. It's the foundation that makes detection possible. Here's the chain from raw log to actionable alert.
Step 1 — Collection
All devices send syslog to a central collector (rsyslog, syslog-ng, Fluentd). UDP port 514 is common; TLS on 6514 for encrypted transport. Your parser reads from this collector or from log files on disk.
Step 2 — Parsing (your code)
The raw strings are converted to structured objects. Now you can query "all logs where src_ip = X" or "all logs where severity = critical and app = sshd". Without parsing, all you have is text.
Step 3 — Normalisation
Different platforms call the same field different things. Cisco calls it "Source", Linux calls it "from", Windows calls it "IpAddress". Normalisation maps them all to the same field name so you can search once across all sources.
Step 4 — Enrichment
Add context to the structured log: GeoIP lookup on src_ip, threat intelligence feed check, hostname resolution, user account lookup in AD. This is what separates a basic parser from a real detection platform.
Step 5 — Correlation
One AUTH001 alert is noise. 200 AUTH001 alerts from the same IP against 30 different usernames in 60 seconds is a brute force attack. Correlation adds the time and count dimension to detection.
Step 6 — Response
Correlated alerts trigger SOAR playbooks: block the IP on the firewall, disable the account, create a ticket, page the analyst. Your parser output feeds this entire chain.
Key log sources and what to look for
Linux auth.log
Watch for: repeated "Failed password"
Watch for: "Invalid user" (username guessing)
Watch for: "Accepted password" from new IPs
Watch for: sudo to root outside business hours
Windows Event Log
4625 = Failed logon (track by account + source IP)
4624 = Successful logon (logon type 3 = network)
4720 = Account created (who, when?)
4688 = Process created (parent process matters)
Cisco ASA
302013/302015 = TCP/UDP connections built
302014/302016 = connections torn down
106023 = access list deny
713228 = VPN session established
Palo Alto TRAFFIC
action = "deny" — blocked connections
action = "allow" + unusual port = suspicious
High bytes from internal host = exfil risk
Unknown app category = C2 or tunneling
Where to take the parser next
Add SQLite output — store parsed logs in a database so you can query across sessions with SQL.

Add threshold correlation — count AUTH001 alerts per (src_ip, 60s window) and fire a higher-severity BRUTE_FORCE_CAMPAIGN alert at 10+.

Add GeoIP — use the geoip2 library to tag each src_ip with country, ASN, and city. Logins from unexpected countries are high-value alerts.

Add threat intel — check src_ip against VirusTotal, AbuseIPDB, or a local threat feed. One line: if the IP is known-bad, bump the alert severity to critical.

Add live tail — use Python's watchdog library to monitor a log file in real time instead of batch processing. This is how a real SIEM agent works.
